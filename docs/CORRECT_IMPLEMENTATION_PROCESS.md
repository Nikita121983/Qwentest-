# Как правильно внедрять изменения по анализу чужого кода

**Версия:** 1.0  
**Дата:** 12 апреля 2026 г.  
**Статус:** Живой документ — обновлять по мере накопления опыта

---

## 1. Когда применять

Этот процесс используется когда:
- Анализируется декомпилированный код чужого приложения (MM, MO, и т.д.)
- На основе анализа пишется собственная реализация
- Требуется воспроизвести поведение оригинала

---

## 2. Главные принципы

### 2.1 Копируй МЕХАНИЗМ, не только каркас

| ❌ Неправильно | ✅ Правильно |
|----------------|--------------|
| Взять SendThread но заменить `wait()/notify()` на `while(sleep)` | Сделать `wait()/notify()` как в оригинале |
| Скопировать state machine но убрать FSM переходы | Воспроизвести FSM как есть |
| Взять очередь команд но без синхронизации | Добавить `synchronized`/`Mutex` как в оригинале |

**Правило:** Если в оригинале используется примитив синхронизации — используем ТОЧНО ТАКОЙ ЖЕ. Упрощение = источник багов.

### 2.2 Один коммит — одно изменение

НЕЛЬЗЯ делать массовую переписку в одном коммите.

```
❌ ПЛОХО:
  commit: "rewrite BLE to match MM/MO" (+264/-63 строк)
  → Невозможно понять что сломалось
  → Невозможно откатить часть изменений

✅ ХОРОШО:
  commit 1: switch transport GATT→RFCOMM (+50/-80)
  commit 2: add SendThread with queue (+80/-10)
  commit 3: add state machine (+40/-5)
  commit 4: add timeout timer (+20)
  commit 5: add heartbeat handling (+30)
```

### 2.3 Тестировать ПОСЛЕ каждого коммита

Каждый коммит должен быть рабочим. После каждого:
1. Собрать и запустить
2. Проверить что новое работает
3. Проверить что старое НЕ сломалось
4. Только потом → следующий коммит

### 2.4 ADR → План → Пошаговое выполнение

НЕЛЬЗЯ реализовать весь план за один раз.

```
ADR принят
    ↓
Создать issue с пронумерованными шагами
    ↓
Шаг 1 → выполнен → протестирован → закоммичен → pushed
    ↓
Шаг 2 → выполнен → протестирован → закоммичен → pushed
    ↓
...
```

## 3. Пошаговый процесс

### Фаза 1: Анализ (Исследование)

```
1. Найти и декомпилировать оригинал
2. Определить ключевые файлы/классы
3. Проследить полный flow от начала до конца:
   - Как начинается подключение?
   - Какие состояния проходят?
   - Как отправляются данные?
   - Как обрабатываются ответы?
   - Как обрабатываются ошибки?
4. Зафиксировать ВСЕ механизмы синхронизации
5. Составить таблицу сравнения
```

**Артефакт:** Документ анализа (типа `docs/24_MM_MO_PROTOCOL_DEEP_ANALYSIS.md`)

### Фаза 2: План (Проектирование)

```
1. На основе анализа создать ADR (если решение архитектурное)
2. Создать issue с конкретными шагами, КАЖДЫЙ шаг = один коммит
3. Приоритизировать: P0 → P1 → P2
4. Для каждого шага определить КРИТЕРИЙ ГОТОВНОСТИ:
   - Что должно работать?
   - Как проверить?
   - Какие логи смотреть?
```

**Шаблон issue:**
```markdown
## Шаг N: [Название]

**Что сделать:** [описание]
**Критерий готовности:** [конкретный проверяемый результат]
**Как проверить:** [конкретные шаги теста]
**Какие логи:** [что должно появиться в логах]
**Затрагиваемые файлы:** [список]
```

### Фаза 3: Реализация (Пошаговая)

Для КАЖДОГО шага:

```
1. Прочитать ОРИГИНАЛЬНЫЙ код ещё раз (не по памяти!)
2. Написать код, ТОЧНО воспроизводя механику
3. НЕ упрощать! Если оригинал сложный — значит так надо
4. Запустить тест/проверку
5. Если работает → коммит
6. Если НЕ работает → НЕ переходить к следующему шагу!
```

**Правило блокировки:** Если шаг N не работает — шаги N+1, N+2 НЕ НАЧИНАТЬ.

### Фаза 4: Верификация

```
1. После каждого коммита:
   - Проверить что новое работает
   - Проверить что старое не сломалось
   - Посмотреть логи
2. После всех шагов:
   - Полное сквозное тестирование
   - Сравнение с оригиналом (работает так же?)
3. Запушить
```

## 4. Чек-лист перед коммитом

Перед КАЖДЫМ коммитом при внедрении чужого кода:

- [ ] Код воспроизводит оригинальную механику (не упрощён)
- [ ] Синхронизация потоков как в оригинале
- [ ] Обработка ошибок как в оригинале
- [ ] Таймауты/retry как в оригинале
- [ ] Запущено и проверено локально
- [ ] Логи показывают ожидаемое поведение
- [ ] Предыдущий функционал НЕ сломан
- [ ] Коммит маленький (один логический блок)
- [ ] Сообщение коммита чёткое: "add X (matching MM/MO)"

## 5. Анти-паттерны (Чего НЕ делать)

| Анти-паттерн | Почему плохо | Пример из проекта |
|--------------|--------------|-------------------|
| **One Big Commit** | Невозможно отладить и откатить | `6791fa1` (+264/-63 за раз) |
| **Упрощение синхронизации** | Race conditions | `while(sleep)` вместо `wait()/notify()` |
| **Копирование без понимания** | Пропущены критичные детали | SendThread без монитора |
| **Всё за один проход** | Нет промежуточной проверки | RFCOMM+SendThread+StateMachine+Heartbeat за раз |
| **План без критериев** | Непонятно когда готово | ADR-013 без конкретных тестов |
| **"Потом доделаю"** | Потом забывается | CRC валидация не добавлена |

## 6. Шаблоны

### 6.1 Шаблон анализа

```markdown
# Анализ [Продукт] — [Компонент]

## Flow
1. [Шаг 1] → [Класс/Метод] → [Что делает]
2. [Шаг 2] → ...

## Механизмы синхронизации
| Место | Механизм | Зачем |
|-------|----------|-------|

## Таблица сравнения
| Элемент | Оригинал | Наша реализация | Совпадает? |
|---------|-----------|-----------------|------------|

## Критичные детали (НЕ УПРОЩАТЬ)
1. ...
2. ...
```

### 6.2 Шаблон issue для пошагового внедрения

```markdown
## [P0/P1/P2] Шаг N: [Название]

**Контекст:** Что делаем и зачем
**Оригинал:** [Файл в MM/MO] → [Класс] → [Метод]
**Что скопировать:**
- [Механизм 1] — точно как в [файл:строка]
- [Механизм 2] — точно как в [файл:строка]

**Критерий готовности:**
- [ ] [Конкретный проверяемый результат]
- [ ] [Логи должны показать X]

**Файлы:** [Список изменяемых файлов]

**Риски:** [Что может сломаться]
```
