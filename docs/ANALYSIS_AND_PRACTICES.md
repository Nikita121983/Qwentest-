## 📊 Анализ прошлого проекта

### Проект

**Назначение:** Программа для обработки файлов фотополимерной 3D печати (LCD/SLA).

**Функция:** Объединение N избыточных срезов для вертикального сглаживания слоёв.

---

### ✅ Что было правильно

| Практика | Оценка | Комментарий |
|----------|--------|-------------|
| Разделение по файлам | ✅ Правильно | Каждый алгоритм в своём файле |
| Разделение UI и логики | ✅ Правильно | Главное окно формирует GUI из файлов алгоритмов |
| Максимальное разделение по логике | ✅ Правильно | Принцип единственной ответственности |

---

### ❌ Что было неправильно

| Проблема | Последствия | Решение |
|----------|-------------|---------|
| **Плохое планирование** | Архитектура менялась в процессе, потеря контроля | ТЗ → Архитектура → Код |
| **Отсутствие тестов** | Страх менять код, баги у пользователя, невозможность рефакторинга | Тесты на ядро — обязательно |
| **Сильное разделение кода** | 50 файлов по 10 строк, сложно понять логику | Разумное разделение: 100-500 строк на файл |

## 🏗️ Новые практики

### 1. Планирование обязательно

**Порядок:**
```
1. ТЗ (TZ_TEMPLATE.md)
2. Архитектура (ARCHITECTURE_TEMPLATE.md)
3. План (PROJECT_CHECKLIST.md)
4. Код (с тестами)
```

**Запрещено:** Начинать писать код без заполненного ТЗ.

---

### 2. Разумное разделение кода

**Критерии:**

| Параметр | Минимум | Норма | Максимум |
|----------|---------|-------|----------|
| Строк на файл | 100 | 200-400 | 500 |
| Параметров на функцию | — | ≤5 | 7 |
| Сложность функции | — | ≤10 | 20 |

**Правило:** Группировать по функциональности, не по принципу «1 файл = 1 функция».

---

### 3. Тесты — обязательно

**Покрытие:**

| Модуль | Покрытие | Тип тестов |
|--------|----------|------------|
| Core (ядро) | 100% | Unit |
| IO (ввод-вывод) | 80% | Unit |
| GUI (интерфейс) | 50% | Integration |
| Utils (утилиты) | 80% | Unit |

**Минимум:**
- Тесты на базовый сценарий
- Тесты на граничные случаи (пустые данные, максимум)
- Тесты на ошибки (невалидный ввод)

---

### 4. Архитектурные слои

```
GUI (Presentation)
  ▼
IO (Application)
  ▼
Core (Domain)
  ▼
Utils (Infrastructure)
```

**Правило:** Зависимости только вниз.

---

## 📋 Рабочий процесс

### Начало проекта

1. Заполнить `TZ_TEMPLATE.md`
2. Адаптировать `ARCHITECTURE_TEMPLATE.md`
3. Пройти `PROJECT_CHECKLIST.md` (этапы 1-3)
4. Создать структуру проекта
5. Настроить инструменты (ruff, pytest, mypy)
6. Написать первый тест

### Начало сессии

1. Открыть `PROJECT_CHECKLIST.md`
2. Посмотреть текущий этап
3. Прочитать последний `*_SUMMARY.md`
4. Начать работу

### Конец сессии

1. Запустить тесты: `pytest --cov=src --cov-fail-under=80`
2. Запустить линтер: `ruff check src/`
3. Сохранить диалог (SUMMARY + FULL опционально)
4. Обновить `PROJECT_CHECKLIST.md` (отметить выполненное)
5. Закоммитить изменения

## ⚠️ Антипаттерны (избегать)

| Антипаттерн | Проблема | Решение |
|-------------|----------|---------|
| «Напишу код, потом тесты» | Тесты никогда не будут написаны | TDD: тест → код → рефакторинг |
| «Сделаю потом документацию» | Документация никогда не будет написана | Сначала ТЗ, потом код |
| «Вынесу в отдельный файл» | 50 файлов по 10 строк | Разумное разделение: 100-500 строк |
| «Добавлю параметр в функцию» | >5 параметров — нечитаемо | Использовать dataclass/config |
| «Зависимость в обратную сторону» | Круговые зависимости | Зависимости только вниз |
