**Дата:** 9 апреля 2026 г.  
**Версия:** 1.0  
**Статус:** Утверждена к разработке

## 1. Роли и Инструменты

### 1.1. Телефон (Android) — Основной инструмент
*   **Пользователь:** Замерщик (Ты).
*   **Задачи:**
    *   Приём потока измерений с рулетки (BLE).
    *   Автоматическая привязка к дверям (Сценарий / PDF-зоны).
    *   Обогащение данных: Маркировка, Контекст проёма (Пол-Потолок, Стена-Стена).
    *   **Предварительный анализ:** Вычисление вариантов решений (Проект / Факт / Компромисс).
    *   **Генерация отчёта:** Создание "Умного Excel" файла для менеджеров.

### 1.2. ПК (Python/Excel) — Вспомогательный инструмент
*   **Пользователь:** Инженер / Аналитик (Ты).
*   **Задачи:**
    *   Управление справочниками (типы дверей, допуски).
    *   Глубокий анализ (сравнение с предыдущими замерами).
    *   Разметка PDF-планов (через Xodo → аннотации).
    *   Резервная обработка данных, если что-то упущено на телефоне.

### 1.3. Excel (Менеджерский файл) — Инструмент принятия решений
*   **Пользователь:** Менеджер / Проектировщик.
*   **Задачи:**
    *   Получение данных в готовом виде (включая варианты решений).
    *   Выбор решения из утвержденного списка (без ручного ввода).
    *   Формирование итоговой спецификации на заказ/производство.

## 2. Структура Файла Выходных Данных (Smart Excel)

Файл (например, `Юнтолово 8 корпус...xlsx`) состоит из стандартных листов (как сейчас) + **Нового листа решений**.

### 2.1. Стандартные листы (Существующие)
1.  **Замер:** Полные данные с подписями (Сырые значения, Время, Тип).
2.  **Дефектная ведомость:** Список дверей с отклонениями от проекта.
3.  **Шахматка / Сводная:** Визуализация по этажам/квартирам.

### 2.2. Лист "Решения" (Новый функционал)
Добавляется для автоматизации выбора размеров заказа.

| Дверь | Контекст | Факт (Ш/В) | Проект (Ш/В) | Отклонение | **Вариант А** | **Вариант Б** | **Вариант В** | **Решение (Выбор)** | **Размер в Заказ** |
|:---|:---|:---|:---|:---|:---|:---|:---|:---|:---|
| Д-1 | Пол→Потолок | 900 / 2100 | 900 / 2100 | 0 | 900/2100 | 900/2100 | 900/2100 | [Авто] | 900 / 2100 |
| Д-2 | Стена→Стена | 880 / 2080 | 900 / 2100 | -20 / -20 | 900 / 2100 | 880 / 2080 | 890 / 2090 | [Вариант Б ▼] | 880 / 2080 |
| Д-3 | Пол→Потолок | 900 / 2050 | 900 / 2100 | 0 / -50 | ❌ Невозм. | ✅ 900/2050 | ⚠️ 900/2080 | [Вариант Б ▼] | 900 / 2050 |

**Ключевые особенности листа "Решения":**
1.  **Предварительный расчет:** Все варианты (А, Б, В) просчитаны **до** открытия файла менеджером (на телефоне или ПК).
2.  **Оценка реализуемости:**
    *   ✅ Возможно сразу.
    *   ⚠️ Возможно с работами (штробление, подливка).
    *   ❌ Невозможно (физические ограничения проёма).
3.  **Защита от ошибок:** Колонка **"Решение"** содержит выпадающий список (Dropdown). Ручной ввод произвольных чисел заблокирован (Protection).
4.  **Автоподстановка:** Колонка **"Размер в Заказ"** автоматически заполняется формулой (`ЕСЛИ(Решение="Б"; Факт; Проект)`).

## 3. Логика Генерации Вариантов (Rules Engine)

Система анализирует каждый проём и генерирует 3 стратегии.

### 3.1. Контекст проёма (Критически важно)
Определяется на этапе замера. Влияет на возможность расширения.
*   `FLOOR_TO_CEILING` (Пол → Потолок): Увеличить высоту **нельзя**.
*   `WALL_TO_WALL` (Стена → Стена): Увеличить ширину **можно** (штробление).
*   `SLAB_TO_SCREED` (Плита → Стяжка): Увеличить высоту **можно** (снятие стяжки).

### 3.2. Генерация Вариантов
Для каждой двери формируются:
1.  **Вариант А (По проекту):** Идеальный размер.
    *   *Проверка:* Если отклонение > допуска AND контекст "Нельзя исправить" → **Пометка "Невозможен"**.
2.  **Вариант Б (По факту):** То, что есть сейчас.
    *   *Статус:* Всегда возможен (заказываем дверь под текущий проём).
3.  **Вариант В (Оптимальный):** Компромисс.
    *   *Логика:* Округление до стандарта или устранение мелкого дефекта.
    *   *Пример:* Если отклонение -50мм, но можно снять стяжку → предложить 2100 с пометкой "Снять стяжку".

## 4. Пайплайн Данных (Data Flow)

```
[ ЗАМЕР (Android) ]
    │
    ├── 1. Приём данных BLE (Raw)
    ├── 2. Определение контекста (Пол/Стена/Стяжка)
    ├── 3. Автомаркировка (Генерация подписи)
    │
    ▼
[ ОБРАБОТКА (Engine) ]
    │
    ├── 4. Сравнение с Проектом (Справочники)
    ├── 5. Расчет отклонений и вариантов (Rules Engine)
    ├── 6. Кластеризация (поиск групп брака)
    │
    ▼
[ ЭКСПОРТ (Smart Excel) ]
    │
    ├── Листы: Замер / Дефектная / Шахматка (Как было)
    └── Лист: Решения (Новое: Варианты + Защита)
    │
    ▼
[ МЕНЕДЖЕР (Excel) ]
    │
    ├── Выбор решения из списка (Без ошибок)
    └── Получение финальной цифры для заказа
```

## 5. Технический стек

| Компонент | Технология | Примечание |
|:---|:---|:---|
| **Телефон (Сбор)** | Android (Kotlin), BLE API, Room DB | Основной инструмент ввода |
| **Обработка** | Python (Pandas, OpenPyXL) | На ПК / Встроенная логика в Phone |
| **Справочники** | SQLite / JSON файлы | Типы дверей, допуски, проекты |
| **Экспорт** | OpenPyXL / Apache POI | Генерация XLSX с защитой и формулами |
| **Разметка** | Xodo PDF + PdfBox-Android | Чтение зон дверей из файлов |
