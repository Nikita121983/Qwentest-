## 3. ПРОЦЕСС ЗАМЕРА

### 3.1. Автоматическое создание Measurement

**Поток:**
```
1. Рулетка отправляет измерение по BLE
2. Приложение получает: 900 мм + timestamp
3. Автоматически создаётся Measurement:
   {
     "measurement_id": "m_20260402_143000_001",
     "timestamp": "2026-04-02T14:30:00",
     "device": "Bosch GLM 50 C",
     "points": []
   }
4. Пользователь видит: "Новый замер: 900 мм"
```

### 3.2. Добавление точек замера

**Стандартная группа (6 замеров на 1 дверь):**

| № | Точка | Значение | Маркировка |
|---|-------|----------|------------|
| **p1** | Ширина (низ) | 1014.5 мм | ✅ 31 П1.1.Кв1... |
| **p2** | Ширина (середина) | 1010.6 мм | — |
| **p3** | Ширина (верх) | 999.3 мм | — |
| **p4** | Высота (левая) | 2208.4 мм | Дп |
| **p5** | Высота (середина) | 2227.1 мм | — |
| **p6** | Высота (правая) | 2228.9 мм | — |

**Алгоритм группировки:**
```
1. Найти строку с маркировкой в CSV
2. Взять предыдущие 5 строк
3. Сгруппировать в 1 дверь (6 точек)
4. Присвоить object_id из маркировки
```

### 3.3. Сверка с планом замера (для типовой планировки)

**Если загружен план замера:**

**Алгоритм сверки:**
```kotlin
fun verifyWithPlan(
    actualMarking: String?,
    expectedDoors: List<ExpectedDoor>,
    currentDoorIndex: Int
): VerificationResult {
    val errors = mutableListOf<String>()
    val warnings = mutableListOf<String>()

    // Проверка наличия маркировки
    if (actualMarking == null) {
        warnings.add("⚠️ Отсутствует маркировка (ожидалась: ${expectedDoors[currentDoorIndex].marking})")
        return VerificationResult(errors, warnings)
    }

    // Проверка соответствия маркировки
    val expectedMarking = expectedDoors[currentDoorIndex].marking
    if (actualMarking != expectedMarking) {
        errors.add("❌ Маркировка не совпадает:")
        errors.add("   Ожидалась: $expectedMarking")
        errors.add("   Фактическая: $actualMarking")
    }

    // Проверка пропуска дверей
    if (currentDoorIndex > expectedDoors.size) {
        errors.add("❌ Пропущены двери! Сделано замеров: $currentDoorIndex, Ожидалось: ${expectedDoors.size}")
    }

    // Проверка завершения
    if (currentDoorIndex == expectedDoors.size) {
        warnings.add("✅ Все двери замерены!")
    }

    return VerificationResult(errors, warnings)
}
```

**Пример уведомления:**
```
✅ Дверь 1/10: Маркировка совпадает (31л П1.1.Кв1)
⏳ Следующая: Дверь 2/10 (31л П1.1.Кв1 санузел)
```

**Пример предупреждения:**
```
⚠️ Маркировка не совпадает!
   Ожидалась: 31л П1.1.Кв1
   Фактическая: 31л П1.1.Кв2

   Пропустить? [Да] [Нет, исправить]
```

### 3.4. Определение типа замера

**Алгоритм:**
```kotlin
fun determineMeasurementType(
    value: Double,
    context: List<Measurement>,
    marking: String?
): MeasurementType {
    return when {
        // Есть маркировка двери
        marking != null -> MeasurementType.DOOR_OPENING

        // Значение 800-1200 мм → ширина двери
        value in 800.0..1200.0 -> MeasurementType.DOOR_WIDTH

        // Значение 2000-2300 мм → высота двери
        value in 2000.0..2300.0 -> MeasurementType.DOOR_HEIGHT

        // Значение > 2500 мм → панорамная
        value > 2500.0 -> MeasurementType.PANORAMIC

        // Значение < 600 мм → возможно окно
        value < 600.0 -> MeasurementType.WINDOW

        else -> MeasurementType.UNKNOWN
    }
}

enum class MeasurementType {
    DOOR_OPENING,      // Проем двери
    DOOR_WIDTH,        // Ширина двери
    DOOR_HEIGHT,       // Высота двери
    PANORAMIC,         // Панорамная дверь
    WINDOW,            // Окно
    UNKNOWN            // Не определено
}
```

## 4. ОБОГАЩЕНИЕ ДАННЫХ (ПАРАЛЛЕЛЬНЫЕ ПРОЦЕССЫ)

### 4.1. Пользовательский ввод

**Ветвь 1: Голосовые заметки**
```
1. Пользователь нажимает "🎤 Записать"
2. Запись аудио (5-30 сек)
3. Сохранение: audio_001.mp3
4. Добавление в очередь на распознавание
5. (Параллельно) Пользователь продолжает замеры
```

**Ветвь 2: Фото**
```
1. Пользователь нажимает "📷 Фото"
2. Камера → фото
3. Сохранение: photo_001.jpg
4. (Опционально) Аннотация (рисунок на фото)
5. Привязка к measurement_id
```

**Ветвь 3: Ручной ввод**
```
1. Пользователь открывает measurement
2. Вводит текст в поле notes_raw
3. Сохранение в БД
```

### 4.2. Автоматическое распознавание аудио

**Параллельный процесс (не блокирует замеры):**

```
1. Аудио добавляется в очередь (Channel)
2. Whisper распознаёт (фон, 10-30 сек)
3. Применение словаря сокращений
4. Запись в notes_processed (отдельный столбец)
5. Уведомление: "✅ Распознано: ЛП слом"
```

**Структура БД:**
```sql
CREATE TABLE Measurements (
    measurement_id TEXT PRIMARY KEY,
    notes_raw TEXT,         -- Пользователь ввёл
    notes_processed TEXT,   -- Авто: распознавание аудио
    notes_analysis TEXT,    -- Авто: анализ паттернов
    notes_label TEXT,       -- Авто: определение двери
    notes_validation TEXT   -- Авто: валидация
);
```

### 4.3. Анализ паттернов

**Параллельный процесс:**

```
1. Чтение последних N замеров
2. Поиск паттернов:
   - 3 замера ширины → группа
   - 3 замера высоты → группа
   - Маркировка → определение двери
3. Определение соответствия:
   - Это дверь №31 на этаже 2
   - Это ширина (не высота)
   - Это проём (не стена)
4. Запись в notes_analysis
```

**Пример:**
```
Входные данные:
  p1: 1014.5 мм (маркировка: 31 П1.1.Кв1)
  p2: 1010.6 мм
  p3: 999.3 мм
  p4: 2208.4 мм (Дп)
  p5: 2227.1 мм
  p6: 2228.9 мм

notes_analysis:
  "Дверь №31, Подъезд 1, Секция 1, Квартира 1
   Ширина: 999-1015 мм (разброс 15.2 мм)
   Высота: 2208-2229 мм (разброс 20.5 мм)
   Тип: Стандартная дверь (1000×2200)"
```

### 4.4. Контроль правильности замера

**Параллельный процесс:**

```
1. Валидация размеров:
   - Ширина < 600 мм → ❌ Ошибка
   - Высота < 1800 мм → ❌ Ошибка
   - Разброс > 20 мм → ⚠️ Предупреждение
2. Проверка геометрии:
   - Диагональ = √(ширина² + высота²) ± 10 мм
3. Проверка комплектности:
   - 6 замеров на дверь → ✅
   - 5 замеров → ⚠️ Неполная группа
4. Запись в notes_validation
```

**Пример:**
```
notes_validation:
  "✅ Группа полная (6 замеров)
   ✅ Маркировка найдена
   ⚠️ Разброс ширины: 15.2 мм (допустимо)
   ⚠️ Разброс высоты: 20.5 мм (допустимо)
   ✅ Рекомендуемые размеры: 954.5 × 2178.4 мм"
```

    - Дверь 4 (входная)

### 6.3. Обратный порядок в CSV

**Проблема:** Bosch CSV отсортирован в обратном порядке (новые замеры сверху)

**Решение:**
```kotlin
fun importBoschCSV(file: File): List<Measurement> {
    val lines = file.readLines()

    // Перевернуть порядок (снизу вверх — от старых к новым)
    val reversedLines = lines.reversed()

    // Сгруппировать по 6 замеров на дверь
    val measurements = groupByDoor(reversedLines)

    return measurements
}
```

## 7. ТЕРМИНОЛОГИЯ

### 7.1. Основные термины

| Термин | Определение |
|--------|-------------|
| **Measurement** | Один замер (одно значение в мм) |
| **Point** | Точка замера (p1, p2, p3...) |
| **Door** | Дверь/проём (группа из 6 замеров) |
| **Floor** | Этаж (группа дверей) |
| **Project** | Объект (группа этажей) |

### 7.2. Разница между созданием Measurement и добавлением Points

**Создание Measurement:**
```
1. Рулетка отправляет измерение
2. Приложение создаёт новую запись:
   {
     "measurement_id": "m_001",
     "timestamp": "...",
     "points": []
   }
```

**Добавление Points:**
```
1. Пользователь группирует замеры
2. Добавляет точки к Measurement:
   {
     "measurement_id": "m_001",
     "points": [
       {"point_id": "p1", "name": "Ширина (низ)", "value": 1014.5},
       {"point_id": "p2", "name": "Ширина (середина)", "value": 1010.6},
       ...
     ]
   }
```

**Итог:**
- **Measurement** = контейнер для замера
- **Points** = конкретные точки внутри замера
