## Архитектурное решение

### Debug-окно для тестирования BLE

Создаётся **универсальное** окно отладки `BleDebugActivity` которое:
- Показывает сырые HEX данные приёма/передачи
- Показывает состояние state machine
- Показывает статус очередей и таймеров
- Позволяет отправлять сырые команды
- Логгирует всё на экран
- **Абстрагировано от Bosch** — работает с любым BLE/RFCOMM устройством

## Пошаговый план

### Шаг 1: Универсальный RFCOMM менеджер (BleConnectionManager)

**Что:** Вынести RFCOMM подключение в отдельный универсальный класс
**Зачем:** GlmBleManager не должен управлять сокетами напрямую
**Критерий готовности:**
- [ ] BleConnectionManager создаётся, подключается, отключается
- [ ] Читает данные из InputStream и передаёт через callback
- [ ] Пишет данные в OutputStream
- [ ] Логи: "Connect OK", "Read N bytes", "Write N bytes"
**Файлы:** `ble/BleConnectionManager.kt` (НОВЫЙ)

---

### Шаг 2: Синхронизация состояний (wait/notify вместо sleep)

**Что:** Заменить polling на `synchronized` + `wait()/notifyAll()`
**Зачем:** Race condition между SendThread и readLoop
**Критерий готовности:**
- [ ] `stateLock` объект для синхронизации
- [ ] `transitionTo()` метод с notifyAll()
- [ ] `waitForState()` метод с wait() и timeout
- [ ] В логах: корректные переходы состояний, нет гонок
**Файлы:** `ble/GlmBleManager.kt` (изменения)

---

### Шаг 3: Потоко-безопасный буфер приёма

**Что:** Добавить `synchronized` доступ к `readBuffer`
**Зачем:** readLoop пишет, processBuffer читает — race condition
**Критерий готовности:**
- [ ] `bufferLock` объект
- [ ] readLoop пишет внутри synchronized
- [ ] processBuffer читает внутри synchronized
- [ ] Нет потерь данных при стресс-тесте
**Файлы:** `ble/GlmBleManager.kt` (изменения)

---

### Шаг 4: CRC валидация при приёме

**Что:** Использовать `parseWithCrc()` вместо `parse()`
**Зачем:** Битые пакеты проходят как валидные
**Критерий готовности:**
- [ ] processBuffer() вызывает parseWithCrc()
- [ ] При CRC mismatch — лог "CRC mismatch: expected=X, got=Y"
- [ ] Битые пакеты НЕ попадают в БД
**Файлы:** `ble/GlmBleManager.kt` (изменения)

---

### Шаг 5: Ожидание ответа после команды

**Что:** SendThread ждёт ответ (измерение или heartbeat) перед следующей командой
**Зачем:** Команды не должны отправляться когда рулетка занята
**Критерий готовности:**
- [ ] После отправки — waitForResponse(500ms)
- [ ] Timeout → лог "Command timeout", retry или drop
- [ ] Ответ получен → next command
**Файлы:** `ble/GlmBleManager.kt` (изменения)

---

### Шаг 6: Debug-окно (BleDebugActivity)

**Что:** Activity с логами, состоянием, кнопками отправки команд
**Зачем:** Визуальная проверка всей отладочной информации
**Критерий готовности:**
- [ ] Activity открывается из меню
- [ ] Показывает: статус подключения, state machine, очередь команд
- [ ] Показывает: сырые HEX RX/TX
- [ ] Кнопка "Send Raw Command" с вводом HEX
- [ ] Авто-скролл логов
**Файлы:** `ui/BleDebugActivity.kt` (НОВЫЙ), `activity_ble_debug.xml` (НОВЫЙ)

---

### Шаг 7: Интеграция Debug-окна и тестирование

**Что:** Подключить BleDebugActivity к BleConnectionManager
**Зачем:** Проверить всё в сборке
**Критерий готовности:**
- [ ] APK собирается
- [ ] Debug-окно открывается
- [ ] Подключение к рулетке → видно в debug-окне
- [ ] Измерение → видно HEX и распарсенные данные
- [ ] Тип/точка отсчёта переключаются
**Файлы:** Множественные мелкие правки

## Инструкция для тестирования

### Что нужно сделать перед тестом:
1. Включить Bluetooth на телефоне
2. Включить рулетку Bosch GLM 50C
3. **На рулетке:** просто включить (кнопка ON) — ничего нажимать НЕ нужно
4. Открыть приложение → меню → "Connect to Ruler" → выбрать рулетку
5. После подключения → меню → "BLE Debug" (новый пункт)

### Что проверять в debug-окне:
- Статус подключения: "Connected"
- State machine: корректные переходы
- RX/TX логи: HEX пакеты совпадают с ожидаемыми
- Измерения: появляются при нажатии кнопки на рулетке
- Команды: Set Type, Set Ref — рулетка реагирует
