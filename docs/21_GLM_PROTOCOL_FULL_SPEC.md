## 2. Формат команд (Host → Device)

Команды отправляются в характеристику `02A6C0D1...` (Write With Response).

**Источники:** `MtFrameByteWriter.java`, `EDCFrameFactory.java`, `Crc.java` из APK MeasureOn.

### 2.1. Три формата фреймов (MtRequestFrame)

#### LONG формат (ui2RequestFrameFormat = 0)
```
┌──────────┬──────────┬────────────┬────────────────┬────────┐
│FrameMode │  Command │ PayloadLen │    Payload     │  CRC8  │
│  1 байт  │  1 байт  │   1 байт   │     N байт     │ 1 байт │
└──────────┴──────────┴────────────┴────────────────┴────────┘
    0xC0      0x55      N (0-255)   Данные команды    poly=0xA6
                                                            Init=0xAA
```
CRC8 вычисляется по `[FrameMode, Command, PayloadLen, Payload...]`

#### SHORT формат (ui2RequestFrameFormat = 1)
```
┌──────────┬──────────┬────────┐
│FrameMode │  Command │  CRC8  │
│  1 байт  │  1 байт  │ 1 байт │
└──────────┴──────────┴────────┘
    0xC1      0x55    poly=0xA6
```
CRC8 вычисляется по `[FrameMode, Command]`

#### EXT формат (ui2RequestFrameFormat = 2)
```
┌──────────┬──────────┬────────┬────────┬────────────────┬─────────┐
│FrameMode │  Command │ LenLSB │ LenMSB │    Payload     │  CRC16  │
│  1 байт  │  1 байт  │ 1 байт │ 1 байт │     N байт     │  2 байт │
└──────────┴──────────┴────────┴────────┴────────────────┴─────────┘
    0xC2      0x55      LSB       MSB     Данные команды
```
CRC16 вычисляется по `[FrameMode, Command, LenLSB, LenMSB, Payload...]`

### 2.2. FrameMode байт (UnionFrameMode)

8 бит = 4 поля по 2 бита:

| Биты | Поле | Размер | Значение |
|------|------|--------|----------|
| **0-1** | ui2FrameType | 2 | `3` = request, `0` = response |
| **2-3** | ui2Reserved | 2 | Зарезервировано |
| **4-5** | ui2RequestFrameFormat | 2 | `0`=LONG, `1`=SHORT, `2`=EXT |
| **6-7** | ui2ResponseFrameFormat | 2 | Формат ожидаемого ответа |

Пример: `0xC0` (192 = `0b11000000`) = request + LONG request + LONG response

### 2.3. CRC8 алгоритм

**Источник:** `Crc.java`

| Параметр | Значение |
|----------|----------|
| **Полином** | `0xA6` (в коде: `(byte)-90`) |
| **Init** | `0xAA` (170 = `EN_CRC8_INITIAL_VALUE`) |
| **Алгоритм** | Побитовый (8 итераций на байт) |

```java
public static byte calcCrc8(byte b, byte b2, byte b3) {
    for (int i = 0; i < 8; i++) {
        b2 = (byte) (((b2 & 0x80) != 0) != (((b >>> (7 - i)) & 1) == 1)
            ? (b2 << 1) ^ b3    // XOR с полиномом 0xA6
            : b2 << 1);
    }
    return b2;
}
```

**Примеры расчёта:**
- `[C0, 55, 02, 01, 00]` → CRC = `0x1A` → Итог: `C0 55 02 01 00 1A AC` (AC = BLE терминатор Android)
- `[C0, 56, 01, 00]` → CRC = `0x1E` → Итог: `C0 56 01 00 1E AC`
- `[C0, 55, 02, 40, 00]` → CRC = `0x70` → Итог: `C0 55 02 40 00 70`

> **Примечание:** Байт `0xAC` в конце HCI логов — это BLE терминатор Android стека, не часть протокола. Приложению его отправлять НЕ нужно.

### 2.4. Подтверждённые команды

| Команда | Hex | Описание |
|---------|-----|----------|
| **Init (AutoSync ON)** | `C0 55 02 40 00 [CRC8]` | Вкл. автосинхронизации (devMode=0, syncCtrl=1) |
| **Set Dev Mode** | `C0 55 02 BC [edcMode] [CRC8]` | **Смена типа** (devMode=60, syncCtrl=1, keypadBypass=0) |
| **Set Distance Ref** | `C0 55 02 BE [refLevel] [CRC8]` | **Смена точки отсчёта** (devMode=62) |
| **Remote Trigger** | `C0 56 01 00 [CRC8]` | Кнопка замера |
| **Sync History** | `C0 55 02 BA [index6\|ind2] [CRC8]` | Запрос элемента истории (devMode=58) |

> **Маппинг смены типа:** remoteCtrlData = `turnSyncModeToEDCMode(syncMode)`
> - syncMode 1→EDC 1 (Direct), 2→4 (Area), 3→7 (Volume), 4→8 (Angle)
> - syncMode 6→2 (Continuous), 7→10 (IndHeight), 8→11 (IndLength), 9→13 (Double), 10→15 (WallArea), 14→26 (Trapezoid)

> **Точки отсчёта:** refLevel 0=Передняя, 1=Штатив, 2=Задняя, 3=Pin

### 2.5. EDCOutputMessage (команды управления EDC)

**Источник:** `EDCOutputMessage.java` + `EDCFrameFactory.java`

Структура (16 бит = 2 байта):
```
[syncControl:1][keypadBypass:1][devMode:6] [remoteCtrlData:8]
```

| Поле | Биты | Описание |
|------|------|----------|
| **syncControl** | 1 | 0=OFF, 1=ON (автосинхронизация) |
| **keypadBypass** | 1 | Обход клавиатуры |
| **devMode** | 6 | Режим измерения (0-63) |
| **remoteCtrlData** | 8 | Дополнительные данные (индекс, индикатор) |

**Пример Sync History (index=5, indicator=0):**
- devMode=58 (0b111010), syncCtrl=1, keypadBypass=0
- Header: `10111010` = `0xBA`
- remoteCtrlData: `(0<<6)|5` = `0x05`
- Итог: `C0 55 02 BA 05 [CRC8]`

### 2.7. KeypadBypass

**Источник:** `EDCOutputMessage.java`, `EDCFrameFactory.java`

1-битное поле в заголовке EDC-фрейма:
- `0` = DISABLE — клавиатура рулетки блокируется (удалённое управление)
- `1` = ENABLE — клавиатура остаётся активной

В Measuring Master всегда `0` (блокировка).

### 2.8. Heartbeat (C0 55 02 F1)

**Статус:** ✅ **РАСШИФРОВАНО** из APK, 3 апреля 2026 г., 20:30 MSK.

`C0 55 02 F1` — это **EDCInputMessage** (команда 85) от рулетки:
- FrameMode = 0xC0 (LONG request/response)
- Command = 0x55 (85 = EDC)
- PayloadLen = 2
- Payload[0] = 0xF1 = DevModeRef: `[syncCtrl=1][keypadBypass=0][devMode=60]`
- Payload[1] = текущий режим рулетки (0-15)

Это **ответ рулетки** на Init команду — она сообщает свой режим.

### 2.9. BLE транспорт (из BLEConnection.java)

**Источник:** `BLEConnection.java`, `MtProtocolBLEImpl.java`

Запись в BLE характеристику:
```java
characteristic.setValue(data);  // data = [FrameMode, Cmd, Len, Payload..., CRC8]
bluetoothGatt.writeCharacteristic(characteristic);
```

**Важно:** Приложение отправляет ТОЛЬКО MtProtocol фрейм (без дополнительных байт).
Байт `0xAC` в HCI логах добавляется BLE стеком, НЕ приложением.

UUID характеристики для записи: `02A6C0D1-0451-4000-B000-FB3210111989`

Подписка на уведомления:
```java
descriptor.setValue(BluetoothGattDescriptor.ENABLE_INDICATION_VALUE);
bluetoothGatt.writeDescriptor(descriptor);
```

### 2.10. SyncList (команда 81)

**Источник:** `SyncListOutputMessage.java`, `SyncListFrameFactory.java`

Пакетный запрос истории. Формат команды:
```
[0xC0][0x51][0x02][indexFrom][indexTo][CRC8]
```
| Байт | Поле | Описание |
|------|------|----------|
| 0 | 0xC0 | FrameMode (LONG) |
| 1 | 0x51 | Command (81 = SyncList) |
| 2 | 0x02 | PayloadLen |
| 3 | indexFrom | Начальный индекс (0-255) |
| 4 | indexTo | Конечный индекс (0-255) |
| 5 | CRC8 | CRC8 по [C0, 51, 02, indexFrom, indexTo] |

**Ответ:** поток SyncInputMessage для каждого индекса от indexFrom до indexTo.

## 3. Формат пакета данных (Device → Host)

Все данные приходят в формате уведомлений (Notifications).

### 3.1. Полная визуальная схема входящего пакета

```
┌─────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────┐
│                                            EDCInputMessage (Device → Host)                                             │
│                                                                                                                         │
│  ┌──────┬──────┬──────┬──────────────┬──────────────┬──────┬──────┬────────────────┬────────────────┬──────────────┐    │
│  │ 0xC0 │ 0x55 │ 0x10 │  DevModeRef  │  DevStatus   │SeqNum│ 0x71 │    Result      │     Comp1      │    Comp2     │    │
│  │1 байт│1 байт│1 байт│   1 байт     │   1 байт     │1 байт│1 байт│   4 байта      │    4 байта     │   4 байта    │    │
│  ├──────┴──────┴──────┴──────────────┴──────────────┴──────┴──────┴────────────────┴────────────────┴──────────────┤    │
│  │                                                   Header (7 байт)                                               │    │
│  ├─────────────────────────────────────────────────────────────────────────────────────────────────────────────────┤    │
│  │                                                Payload (12+ байт)                                               │    │
│  └─────────────────────────────────────────────────────────────────────────────────────────────────────────────────┘    │
│                                                                                                                         │
│  Смещение:  0      1      2       3          4          5      6      7-10         11-14          15-18         │    │
│  Размер:    1      1      1       1          1          1      1       4             4              4            │    │
│                                                                                                                         │
└─────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────┘
```

### 3.2. Детальная расшифровка полей

#### Заголовок (байты 0-6)

| Смещение | Размер | Поле | Значение | Описание |
|----------|--------|------|----------|----------|
| **0** | 1 байт | FrameMode | `0xC0` | LONG request/response |
| **1** | 1 байт | Command | `0x55` | 85 = EDC команда |
| **2** | 1 байт | PayloadLen | `0x10` | 16 байт минимум |
| **3** | 1 байт | **DevModeRef** | `[refEdge:2][devMode:6]` | Режим + точка отсчёта |
| **4** | 1 байт | **DevStatus** | `[laser:1][temp:1][batt:1][units:1][status:4]` | Статус устройства |
| **5** | 1 байт | SeqNum | 0-255 | Номер пакета (растёт) |
| **6** | 1 байт | Константа | `0x71` | Фиксированный маркер |

#### Данные (байты 7-18)

| Смещение | Размер | Поле | Тип | Единица | Описание |
|----------|--------|------|-----|---------|----------|
| **7-10** | 4 байта | **Result** | float32 LE | **метры** | Основной результат замера |
| **11-14** | 4 байта | **Comp1** | float32 LE | **метры** | Для косвенных = `a`, для двойных = `b` |
| **15-18** | 4 байта | **Comp2** | float32 LE | **градусы** | Угол наклона |

#### Дополнительные данные (байты 19+)

| Смещение | Размер | Поле | Тип | Описание |
|----------|--------|------|-----|----------|
| **19-20** | 2 байта | **MeasID** | uint16 LE | ID замера (если доступен) |
| **21+** | ? | **Tail** | байты | Дополнительные метаданные / padding |

### 3.3. DevModeRef (байт 3) — битовая структура

```
┌──────────────────────────────────────────────────────────────┐
│                     DevModeRef (1 байт = 8 бит)              │
│                                                              │
│  ┌─────────────┬───────────────────────────────────────────┐ │
│  │  refEdge:2  │              devMode:6                    │ │
│  │  (биты 0-1) │             (биты 2-7)                    │ │
│  └─────────────┴───────────────────────────────────────────┘ │
│                                                              │
│  Пример: 0x06 = 0b00000110                                  │
│    refEdge = 0b00 = 0 (Задняя грань)                        │
│    devMode = 0b000001 = 1 (Прямой замер)                    │
│                                                              │
│  Пример: 0x2A = 0b00101010                                  │
│    refEdge = 0b10 = 2 (Ось штатива)                         │
│    devMode = 0b101010 = 10 (Косвенная высота)               │
└──────────────────────────────────────────────────────────────┘
```

**Таблица refEdge (точка отсчёта):**

| refEdge | Значение | Константа в APK | Смещение (примерно) |
|---------|----------|-----------------|---------------------|
| `0` | **Задняя грань** | `REF_EDGE_DISTANCE_REAR` | 0 мм |
| `1` | **Ось штатива** | `REF_EDGE_DISTANCE_TRIPOD` | +30-50 мм |
| `2` | **Передняя грань** | `REF_EDGE_DISTANCE_FRONT` | +60-80 мм |
| `3` | **Pin** | `REF_EDGE_DISTANCE_PIN` | Зависит от аксессуара |

> **Важно:** Рулетка отправляет уже исправленное значение (с учётом смещения). Флаг refEdge — информационный, для отображения в UI.

### 3.4. DevStatus (байт 4) — битовая структура

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                          DevStatus (1 байт = 8 бит)                          │

│  ┌──────┬──────┬──────┬──────┬───────────────────────────────────────────┐  │

│  │ 1бит │ 1бит │ 1бит │ 1бит │           (биты 4-7)                      │  │

| **4-7** | **deviceStatus** | Различные статусы | `8` = Ready |

     │  │  │  │  │  │  │          │
     │  │  │  │  │  │  │          └─ Result (байты 7-10)

     │  │  │  │  │  │  │
     │  │  │  │  │  │  └─ 0x71 (константа)
     │  │  │  │  │  │
     │  │  │  │  │  └─ SeqNum = 8
     │  │  │  │  │
     │  │  │  │  └─ DevStatus = 0x08 → laser=0, Ready
     │  │  │  │
     │  │  │  └─ DevModeRef = 0x06 → refEdge=0 (Задняя), devMode=1 (Direct)

## 4. Heartbeat / Статус (Device → Host)

Периодические пакеты состояния.

```
┌──────┬──────┬──────┬──────┬──────┬──────┬──────┐
│ 0xC0 │ 0x55 │ 0x02 │ 0xF1 │ Mode │Angle │ CRC  │
│1 байт│1 байт│1 байт│1 байт│1 байт│1 байт│1 байт│
└──────┴──────┴──────┴──────┴──────┴──────┴──────┘
                               │      │
                               │      └─ Угол наклона (0-255, условные единицы)

|----------|--------|------|----------|
| **0** | 1 байт | `0xC0` | FrameMode |
| **1** | 1 байт | `0x55` | Command |
| **2** | 1 байт | `0x02` | PayloadLen |
| **3** | 1 байт | `0xF1` | DevModeRef: `[syncCtrl=1][keypadBypass=0][devMode=60]` |
| **4** | 1 байт | **Mode** | Текущий режим рулетки (0-15) |
| **5** | 1 байт | **Angle** | Текущий угол наклона (0-255) |
| **6** | 1 байт | **CRC** | Контрольная сумма (почти всегда `0xAC`) |

## 5. Точка отсчёта (Reference Point)

**Статус:** ✅ **РАСШИФРОВАНО** из APK, 3 апреля 2026 г., 20:30 MSK.

### 5.1. Входящие пакеты (Device → Host)

**Источник:** `EDCMessageFactory.java` — `DevModeRef` класс

Точка отсчёта передаётся в байте 3 (DevModeRef), биты 0-1:

| refEdge (биты 0-1) | Точка отсчёта | Константа в APK |
|--------------------|---------------|-----------------|
| `0` | **Задняя грань** | `REF_EDGE_DISTANCE_REAR` |
| `1` | **Ось штатива** | `REF_EDGE_DISTANCE_TRIPOD` |
| `2` | **Передняя грань** | `REF_EDGE_DISTANCE_FRONT` |
| `3` | **Pin** | `REF_EDGE_DISTANCE_PIN` |

> **Важно:** Рулетка отправляет уже исправленное значение (с учётом смещения). Флаг refEdge — информационный.

### 5.2. Исходящие команды (Host → Device)

**Источник:** `RemoteControlCommandsUtils.java`, 3 апреля 2026 г., 19:45 MSK.

- `devMode = 62` (`MODE_SET_DISTANCE_REFERENCE`)
- `syncControl = 1`
- `keypadBypass = 0`

Формат байта: `[syncCtrl:1][keypadBypass:0][devMode:6]` = `0xBE`

### 5.3. Угловая точка отсчёта

Для угла (`devMode = 61`, `MODE_SET_ANGLE_REFERENCE`):

|----------|-------------------|
| `0` | Задняя грань |
| `1` | Боковая грань |
| `2` | Ось штатива |

## 6. Транспортный уровень (BLE)

**Источники:** `MtProtocolBLEImpl.java`, `MtFrameByteReader.java`, `MtFrameConstants.java`
**Дата анализа:** 3 апреля 2026 г., 22:15 MSK.

### 6.1. Состояния протокола

```
┌─────────────────┐       EVENT_SEND        ┌─────────────────┐
│  SLAVE          │ ──────────────────────→ │  MASTER         │
│  LISTENING      │                         │  READY          │
│  (ожидание)     │ ←────────────────────── │                 │
└─────────────────┘   EVENT_RECEIVE_FINISH  └────────┬────────┘
                                                      │
                                          EVENT_SEND_START
                                                      │
                                                      ▼
                                               ┌─────────────────┐
                                               │  MASTER         │
                                               │  RECEIVING      │
                                               │  (ожидание ответа)│
                                               └─────────────────┘
```

- **SLAVE_LISTENING**: Рулетка ждёт команд от хоста
- **MASTER_READY**: Хост готов отправить команду
- **MASTER_RECEIVING**: Хост ждёт ответ от рулетки

### 6.2. Размеры FIFO

| Параметр | Значение |
|----------|----------|
| RX FIFO | 244 байта |
| TX FIFO | 244 байта |

BLE уведомления могут содержать фрагменты фреймов или несколько фреймов. `MtFrameByteReader` пересобирает их байт-за-байтом.

### 6.3. Парсинг входящих фреймов (MtFrameByteReader)

**State Machine:**
```
┌──────┐    ┌────────┐    ┌─────┐    ┌─────────┐    ┌─────────┐    ┌──────┐    ┌──────────┐    ┌──────────┐    ┌──────┐
│ MODE │ →  │ STATUS │ →  │ CMD │ →  │ SIZE_LSB│ →  │ SIZE_MSB│ →  │ DATA │ →  │ CRC_BYTE1│ →  │ CRC_BYTE2│ →  │ DONE │
└──────┘    └────────┘    └─────┘    └─────────┘    └─────────┘    └──────┘    └──────────┘    └──────────┘    └──────┘
```

**FrameMode байт** определяет:
- Тип фрейма (REQUEST=3, RESPONSE=0)
- Формат фрейма (LONG=0, SHORT=1, EXT=2)
- Формат ожидаемого ответа

### 6.4. Статусы ответа (ComStatus)

| Код | Константа | Описание |
|-----|-----------|----------|
| 0 | `EN_COMM_STATUS_SUCCESS` | Успех |
| 1 | `EN_COMM_STATUS_TIMEOUT` | Таймаут |
| 2 | `EN_COMM_STATUS_MODE_NOT_SUPPORTED` | Режим не поддерживается |
| 3 | `EN_COMM_STATUS_CHECKSUM_ERROR` | Ошибка CRC |
| 4 | `EN_COMM_STATUS_CMD_UNKNOWN` | Неизвестная команда |
| 5 | `EN_COMM_STATUS_ACCESS_DENIED` | Доступ запрещён |
| 6 | `EN_COMM_STATUS_PARAM_ERROR` | Ошибка параметров |

### 6.5. Обработка ошибок

- При ошибке записи: сообщение возвращается в начало очереди, пауза 200мс, повтор
- При таймауте: переключение обратно в SLAVE режим
- При CRC ошибке: `MtProtocolFatalErrorEvent`, соединение разорвано

### Формат BLE уведомлений

**Префикс:** `C0 55 10 06`
**Данные:** байты 7-10 = расстояние (float32 little-endian, в метрах)

```
C0 55 10 06 XX XX XX XX YY YY YY YY
  │  │  │  │  └─ байты 7-10: float32 distance (метры)
  │  │  │  └─ байт 6: тип уведомления
  │  │  └─ байт 5: 0x10 = measurement
  │  └─ байт 4: 0x55 = BLE prefix
  └─ байт 3: 0xC0 = start byte
```

**Типы уведомлений (из HCI лога):**
```
C0 55 10 02 → ? (начало измерения)
C0 55 10 06 → distance (результат)
C0 55 10 0A → ?
C0 55 10 0E → ?
C0 55 10 12 → ?
... до C0 55 10 3E
```

### BLE команда (Nordic DevZone)

```
C0 56 01 00 1E
  1-я запись → активация лазера
  2-я запись → запуск измерения
Ответ → float (расстояние)
```

### 7.2. Расшифровка структуры пакета (по паттернам)

На этом этапе мы определили структуру **только по паттернам** в сырых данных:

```
Входящий пакет: [C0][55][10][TT][SS][NN][71][R:4][C1:4][C2:4][Tail...]
```

| Байт | Определение | Как нашли |
|------|-------------|-----------|
| `C0 55 10` | Заголовок (FrameMode+Cmd+Len) | Постоянен во всех пакетах |
| `TT` | Тип замера | Менялся при смене режима, совпадал с TXT |
| `SS` | Статус устройства | Зависел от laser/battery |
| `NN` | Sequence number | Рос с каждым пакетом |
| `71` | Константа/маркер | Всегда `0x71` |
| `R:4` | Результат (float32 LE) | Совпал с числом в TXT экспорте |
| `C1:4` | Компонент 1 (float32 LE) | `a` для косвенных, `NULL` для прямых |
| `C2:4` | Компонент 2 (float32 LE) | Угол `b` для косвенных |

**Определение типов по hex байту TT (до APK):**

| Hex | Тип | Как определили |
|-----|-----|---------------|
| `0x06` | Прямой замер | Совпал с простым расстоянием в TXT |
| `0x2A` | Косвенная высота | Появлялся когда рулетка вверх, 3 компонента |
| `0x2E` | Косвенная длина | Появлялся при боковом замере, 3 компонента |
| `0x36` | Двойная косвенная | 3 компонента, угол отличается |
| `0x3E` | Площадь | Значение × 1000 = мм² (совпало с TXT) |
| `0x1E` | Объём | Значение × 10⁶ = мм³ (совпало с TXT) |
| `0x21` | Угол | Значение в градусах (совпало с TXT) |
| `0x0A` | Непрерывный | Много пакетов подряд с одним типом |
| `0xF2` | Референс/Тест | Странные значения, не замеры |

**Float32 Little Endian:**
- Пакет: `69 00 05 40` → `0x40050069` → `2.0782` (float)
- Проверка: 2.0782 м = 2078.2 мм ✅ (совпало с TXT)

**Пример расшифровки:**
```
Пакет: C0 55 10 06 08 08 71 69 00 05 40 00 ...
                     │  └──────┬──────┘
                     │         │
                     │         └─ Result (float32 LE)
                     │            0x40050069 = 2.0782 м = 2078.2 мм ✅
                     │
                     └─ Тип 0x06 = Прямой замер
```

### 7.3. Ключевая находка: Байт `0xAC`

При парсинге HCI логов обнаружено:
```
Исходящие: C0 55 02 40 00 1A AC
Входящие:  C0 55 10 06 08 08 71 69 00 05 40 00 ...
```

**Байт `0xAC`** в конце исходящих пакетов — это **BLE терминатор Android стека**, НЕ часть протокола рулетки.
