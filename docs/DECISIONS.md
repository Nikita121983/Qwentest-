## ADR-010: Использование Classic Bluetooth (RFCOMM) для Bosch GLM 50C

**Дата:** 6 апреля 2026 г.
**Статус:** Принято

### Контекст
Изначально предполагалось использование BLE GATT (UUID `02A6C0D0...`), так как рулетка обнаруживается через BLE рекламу. Однако попытки подключения через GATT приводили к разрывам или отсутствию данных.
Анализ логов оригинального приложения (*Measuring Master*) показал использование протокола **RFCOMM (Serial Port Profile)**.

### Решение
1.  **Протокол**: Переход на Classic Bluetooth (RFCOMM).
2.  **Канал**: Используется **Channel 5** (через рефлексию `createRfcommSocket(5)`), так как стандартный UUID SPP не обеспечивает стабильное соединение с данной прошивкой.
3.  **Команды управления**:
    *   `Get Settings` (`C0 55 02 3B 00`) работает как команда получения текущего значения с экрана рулетки.
    *   `Init` (`C0 55 02 40 00`) необходим для инициализации сессии.
    *   `Set Mode` / `Set Ref` (`C0 55 02 BC/BE ...`) позволяют менять режимы с телефона.
4.  **Парсинг**:
    *   Рулетка может отправлять пакеты с байтом `0x72` (вместо `0x71`) в заголовке. Парсер обновлен для приема обоих вариантов.
    *   Исправлена критическая ошибка сравнения `toUInt()` (отрицательные байты) на `toUByte()` / `and 0xFF`.

### Результаты
-   ✅ Данные с рулетки успешно принимаются (HEX-логи подтверждают передачу значений).
-   ✅ Управление режимами работает.
-   ⚠️ **Текущая проблема**: Нестабильность соединения на этапе рукопожатия (иногда не подключается). Требуется дальнейшая отладка сокета.

## ADR-009: Архитектура хранения данных — защита от потерь (на основе анализа MeasureOn)

**Дата:** 3 апреля 2026 г., 23:30 MSK
**Статус:** Принято

### Контекст
Анализ кода Bosch MeasureOn (v2.0.1) выявил критические архитектурные недостатки, приводящие к потере данных:
1. Замеры хранятся в отдельных JSON файлах (без транзакций)
2. Удаление группы безвозвратно удаляет файлы замеров
3. Cloud Sync может перезаписать локальные данные при конфликте
4. Экспорт зависит от текущей UI-группы/проекта

### Решение
В GlmReader будет использована следующая архитектура:
1. **SQLite** — единая БД для всех сущностей (замеры, группы, проекты, настройки)
2. **Транзакции** — `BEGIN`/`COMMIT` при сохранении, гарантия целостности
3. **Soft Delete** — флаг `is_deleted` вместо физического удаления, корзина с таймером очистки
4. **Raw Log Table** — таблица `raw_ble_log` с сырыми байтами пакетов `[timestamp, raw_bytes, packet_len]`. Только `INSERT`, никогда `UPDATE`/`DELETE`. Позволяет восстановить данные при сбое парсера.
5. **Независимый экспорт** — экспорт работает напрямую с БД, не зависит от UI-контекста или выбранных групп
6. **Автосохранение** — каждый замер сохраняется в БД сразу после получения и парсинга BLE пакета

### Последствия
- ✅ Исключена потеря данных при краше приложения
- ✅ Возможность восстановления удалённых замеров из корзины
- ✅ Возможность аудита и отладки через Raw Log
- ✅ Надёжный экспорт, не зависящий от состояния UI
- ⚠️ Требуется реализация миграций БД при обновлении схемы

## ADR-008: GlmReader — C# .NET 8 WinForms для Bosch GLM 50C

**Дата:** 3 апреля 2026 г.
**Статус:** В работе

### Контекст
Создано минимальное приложение для подключения к GLM 50C по BLE.

### Решение
- Тип измерения и точка отсчёта выбираются КНОПКАМИ НА РУЛЕТКЕ
- Приложение только получает данные, не отправляет команды смены типа
- Парсинг подтверждён: 12/12 значений совпали с TXT экспортом

### Нерешённые проблемы
1. BLE: Windows не видит сервис GLM без спаривания
2. Команда смены типа: не расшифрована (HCI не захватывает host→controller)
3. Спаривание рулетки в Windows не работает

### Следующие шаги
- Сравнить HCI логи cosine vs app_control для поиска команды смены типа
- Попробовать Python/bleak или Android как мост для BLE

## ADR-007: GitHub интеграция и CI/CD

**Дата:** 2 апреля 2026 г.
**Статус:** Принято
**Автор:** Nik

### Контекст

Требуется настроить автоматизацию для разработки на GitHub.

### Проблема

- Ручная проверка кода перед мержем
- Тесты проходят только локально
- Нет стандартов для Pull Request
- Сложно отслеживать изменения

### Решение

**GitHub интеграция с полным набором:**

1. **GitHub Actions workflows:**
   - `ci.yml` — тесты и сборка при push/PR
   - `validate-pr.yml` — валидация Pull Request

2. **Pre-commit хуки:**
   - ruff (линтер)
   - black (форматтер)
   - mypy (type checker)
   - pytest (тесты)

3. **Шаблоны:**
   - Issue templates (Bug Report, Feature Request)
   - Pull Request template
   - CODEOWNERS (автоматические ревьюверы)

4. **Документы:**
   - CONTRIBUTING.md (руководство контрибьютора)
   - CHANGELOG.md (история изменений)
   - CODE_REVIEW_CHECKLIST.md (чеклист ревью)

### Структура

```
.github/
├── workflows/
│   ├── ci.yml              ← CI/CD пайплайн
│   └── validate-pr.yml     ← Валидация PR
├── ISSUE_TEMPLATE/
│   ├── bug_report.md
│   └── feature_request.md
├── PULL_REQUEST_TEMPLATE.md
└── CODEOWNERS              ← Автоматические ревьюверы
```

### Последствия

**Положительные:**
- Автоматическая проверка каждого коммита
- Стандарты для PR и Issues
- Покрытие тестов отслеживается
- Pre-commit не даёт закоммитить ошибки

**Отрицательные:**
- Нужно настраивать Secrets (Codecov и др.)
- CI/CD требует времени на запуск (~2-5 мин)

## ADR-006: Гибридная система архивирования с выбором места хранения

**Дата:** 2 апреля 2026 г.
**Статус:** Принято
**Автор:** Nik

### Контекст

Проекты могут быть в разных директориях. Требуется гибкая система архивирования диалогов.

### Проблема

- Разные проекты могут быть в разных папках
- Нужно выбирать: архив в проекте или централизованно
- Нужен общий индекс всех проектов

### Решение

**Гибридная система с выбором места хранения:**

1. **Каждый проект выбирает** где хранить архив:
   - `LOCAL` — в проекте (`{Project}/docs/conversations/`)
   - `GLOBAL` — централизованно (`~/.qwen/conversations/{Project}/`)

2. **Файл выбора:** `docs/conversations/ARCHIVE_LOCATION.md`
   - Указывает текущее место хранения
   - Содержит инструкции по изменению

3. **Два уровня индекса:**
   - Локальный: `{Project}/docs/conversations/INDEX.md`
   - Глобальный: `~/.qwen/conversations/INDEX.md`

4. **Глобальный индекс содержит:**
   - Таблицу всех проектов
   - Путь к каждому проекту
   - Место хранения архива
   - Ссылку на локальный индекс

### Структура

```
~/.qwen/
├── QWEN.md                    ← глобальные предпочтения
└── conversations/
    └── INDEX.md               ← глобальный индекс всех проектов

{Project}/
├── .qwen/QWEN.md              ← конфиг проекта
└── docs/
    ├── conversations/
    │   ├── INDEX.md           ← локальный индекс
    │   ├── ARCHIVE_LOCATION.md ← выбор места хранения
    │   ├── 001_..._FULL.md
    │   └── 001_..._SUMMARY.md
    ├── CONVERSATIONS.md       ← хронология
    └── DECISIONS.md           ← решения проекта
```

### Последствия

**Положительные:**
- Гибкость выбора места хранения
- Все проекты в глобальном индексе
- Можно изменить место хранения постфактум
- Проект самодостаточен (при LOCAL)

**Отрицательные:**
- Два уровня индекса (сложнее)
- Нужно поддерживать актуальность обоих индексов

## ADR-005: Формат сохранения диалогов

**Дата:** 2 апреля 2026 г.
**Статус:** Принято
**Автор:** Nik

### Контекст

Требуется удобный способ сохранения полных диалогов с Qwen Code.

### Проблема

- Выделение всего диалога мышью неудобно (Ctrl+A не работает в интерфейсе)
- Один файл с混合 содержимым (полный текст + резюме) неудобен для чтения

### Решение

Каждый диалог сохраняется в **два файла**:

1. **`###_YYYY-MM-DD_topic_FULL.md`** — полная копия из интерфейса
   - Заполняет: пользователь (копирование из Qwen Code)
   - Содержит: дословный текст всего диалога

2. **`###_YYYY-MM-DD_topic_SUMMARY.md`** — структурированное резюме
   - Заполняет: Qwen Code
   - Содержит: решения, код, итоги, ключевые моменты

Формат именования: `001_2026-04-02_initial-setup_FULL.md`

### Последствия

**Положительные:**
- Разделение полного текста и резюме
- Удобно искать решения в SUMMARY
- Полная копия сохраняется для контекста
- Простой рабочий процесс

**Отрицательные:**
- Два файла на каждый диалог
- Нужно вручную копировать текст

## ADR-011: Протокол Bosch GLM 50C полностью расшифрован

**Дата:** 3 апреля 2026 г.
**Статус:** Принято
**Автор:** Nik + Qwen Code

### Контекст

Требуется создать собственное Windows-приложение для лазерной рулетки Bosch GLM 50C, так как официальные приложения (Measuring Master, MeasureOn) имеют баги (пропажа данных, неправильное время в экспорте).

### Решение

**Полный реверс-инжиниринг BLE протокола:**

1. **Записано 3 HCI лога** с разными сценариями использования
2. **Сопоставлены HCI логи с TXT экспортами** (100% совпадение значений)
3. **Расшифрованы все типы измерений:**
   - `0x06` — Прямой замер (мм)
   - `0x2A` — Косвенная высота (мм)
   - `0x2E` — Косвенная длина (мм)
   - `0x36` — Двойная косвенная (мм)
   - `0x3E` — Площадь (дм²)
   - `0x1E` — Объём (дм³)
   - `0x21` — Угол (миллиградусы)
4. **Найдены команды управления:**
   - `C0 55 02 01 00 1A AC` — Инициализация
   - `C0 56 01 00 1E AC` — Запуск замера
   - `C0 55 10 3A ...` — Синхронизация истории
5. **Расшифрована структура пакетов:**
   - Header: `C0 55 10 TT SS NN 71`
   - Payload: Result (float32), Component (float32), Angle (float32)
   - Tail: Метаданные (ID, счетчики)

### Последствия

**Положительные:**
- Можно написать собственное приложение без багов
- Полный контроль над данными
- Возможность экспорта в правильном формате

**Отрицательные:**
- Точка отсчёта (Reference Point) не расшифрована
- Нужно экспериментально выяснить формат Tail

## ADR-012: Управление BLE callback'ами — избегать автоподключения

**Дата:** 6 апреля 2026 г.
**Статус:** Принято
**Автор:** Nik + Qwen Code

### Контекст

Приложение GlmReader Android не находило устройства при сканировании BLE, хотя Nordic nRF Connect успешно видел и подключался к рулетке.

### Проблема

Callback `onDeviceFound` в `setupBleCallbacks()` автоматически:
1. Останавливал сканирование при первом найденном устройстве
2. Пытался подключиться к нему

Это мешало диалогу `showBleDialog()` заполнять список устройств для выбора пользователем.

### Решение

**Паттерн управления callback'ами:**
1. **Основной поток:** `onDeviceFound` только логирует, не подключается автоматически
2. **Диалог подключения:** Устанавливает свой callback для заполнения списка устройств
3. **Закрытие диалога:** Восстанавливает logging-only callback, НЕ сбрасывает в null

### Код

```kotlin
// В setupBleCallbacks() - только логирование
bleManager.onDeviceFound = { mac, name ->
    Log.d("BLE", "Found: $name ($mac)")
    // Не подключаемся автоматически
}

// В showBleDialog() - callback для заполнения списка
val scanCallback: (String, String) -> Unit = { mac, name ->
    if (foundDevices.none { it.mac == mac }) {
        foundDevices.add(BleDeviceItem(name, mac, false))
        deviceAdapter.notifyDataSetChanged()
    }
}
bleManager.onDeviceFound = scanCallback

// При закрытии диалога - восстановить logging-only callback
bleManager.onDeviceFound = { mac, name ->
    Log.d("BLE", "Found: $name ($mac)")
}
```

### Последствия

**Положительные:**
- Пользователь видит все найденные устройства и может выбрать
- Нет конфликтов callback'ов между основным потоком и диалогом
- Логирование работает всегда для отладки

**Отрицательные:**
- Нужно вручную управлять callback'ами при открытии/закрытии диалога

## ADR-013: Глубокий анализ протокола MM/MO — команды верны, проблема в транспорте

**Дата:** 6 апреля 2026 г.
**Статус:** Принято
**Задача:** Qwentest-4r6

### Контекст
Проведён досканальный анализ декомпилированного кода Measuring Master 1.9.4 и MeasureOn 2.0.1 для поиска причин нестабильности GlmReaderAndroid.

### Анализ

**Команды протокола — ПОЛНОСТЬЮ СОВПАДАЮТ ✅**
- `Init` → `C0 55 02 40 00 [CRC8]` — байт-в-байт как в MO
- `Set Type` → `C0 55 02 BC [mode] [CRC8]` — совпадает
- `Set Ref` → `C0 55 02 BE [ref] [CRC8]` — совпадает
- `Trigger` → `C0 56 01 00 [CRC8]` — совпадает
- CRC8 (poly=0xA6, init=0xAA) — алгоритм идентичен

**Транспорт — КРИТИЧЕСКИЕ РАСХОЖДЕНИЯ ❌**

| Параметр | MM RFCOMM | MO BLE GATT | GlmReaderAndroid | Проблема |
|----------|-----------|-------------|------------------|----------|
| Метод | `createInsecureRfcommSocketToServiceRecord()` | `connectGatt()` | `createRfcommSocketToServiceRecord()` | Secure socket вместо insecure |
| Bonding | **НЕ делает** | До подключения | **После подключения** | Разрывает сессию |
| SendThread | ✅ Очередь + FSM | ✅ Очередь + FSM | ❌ Прямая отправка | Нет гарантии доставки |
| Write confirmation | ✅ | ✅ flagIsBLEWriteFinished | ❌ Нет | Не знаем, приняты ли данные |
| State machine | ✅ SLAVE/MASTER | ✅ SLAVE/MASTER | ❌ Нет | Команды когда угодно |
| Timeout | ✅ 500ms | ✅ 500ms | ❌ Нет | Бесконечное ожидание |
| Retry | ✅ AutoConnectThread | ✅ 3 попытки | ❌ Нет | Один шанс |
| Heartbeat | ✅ Обрабатывается | ✅ Обрабатывается | ❌ Игнорируется | `C0 55 02 F1` не парсится |

### Решение

**Приоритетные исправления:**
1. Заменить `createRfcommSocketToServiceRecord()` на `createInsecureRfcommSocketToServiceRecord()` (как MM)
2. Убрать bonding после подключения (MM не делает bonding для RFCOMM)
3. Добавить SendThread с очередью команд
4. Добавить state machine (SLAVE_LISTENING → MASTER_READY → SENDING → RECEIVING)
5. Добавить timeout timer (500ms)
6. Добавить обработку heartbeat пакетов
