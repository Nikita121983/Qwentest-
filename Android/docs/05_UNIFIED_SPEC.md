## ЧАСТЬ 1: ОБЩИЕ СВЕДЕНИЯ

### 1.1. Назначение системы

**MeasuringAssistant** — это Android-приложение для профессиональных замеров в строительстве (двери, окна, отделка).

**Основная задача:**
- Приём измерений с лазерной рулетки Bosch GLM 50 C по Bluetooth
- Голосовые заметки с оффлайн-распознаванием
- Привязка замеров к PDF-планам (через Xodo)
- Экспорт в нормальные форматы (XLSX/CSV) **с правильным временем замера**

### 1.2. Проблемы, которые решает

| Проблема | Текущее (Bosch MM) | Решение (MeasuringAssistant) |
|----------|-------------------|------------------------------|
| **Пропажа замеров** | 🔴 Замеры исчезают из БД | ✅ Автосохранение каждого замера |
| **Время в экспорте** | 🔴 Время экспорта, не замера | ✅ Timestamp замера сохраняется |
| **Порядок замеров** | 🔴 Теряется при экспорте | ✅ Хронологический порядок |
| **Голосовые заметки** | ❌ Нет | ✅ Запись + распознавание + словарь |
| **Привязка к PDF** | ⚠️ Ручная | ✅ Xodo + слой аннотаций |
| **Синхронизация** | ⚠️ Облако (нестабильно) | ✅ Syncthing (локально) |

### 1.3. Платформы

| Платформа | Технология | Статус |
|-----------|------------|--------|
| **Android** | Kotlin (Native) | 🔴 Основная (разработка) |
| **Windows** | C# / WPF или Avalonia | 🟡 После определения возможностей Android |

## ЧАСТЬ 3: КЛЮЧЕВЫЕ КОМПОНЕНТЫ

### 3.1. BLE Service (Bosch GLM 50 C)

**Протокол:**
- UUID сервиса: `0000fff0-0000-1000-8000-00805f9b34fb`
- UUID характеристики (RX): `0000fff1-0000-1000-8000-00805f9b34fb`
- UUID характеристики (TX): `0000fff2-0000-1000-8000-00805f9b34fb`

**Формат пакета:**
```
Запрос измерения: C0 40 00 EE
Ответ: 00 04 13 0E 00 00 32

Расшифровка:
- 00 04: длина
- 13: тип (измерение)
- 0E 00 00 32: значение (0x3200000E = 2098 мм)
```

**Ресурсы:**
- https://github.com/.../Bosch-GLM50C-Rangefinder
- https://github.com/.../BOSCH-GLM-rangefinder

### 3.2. Whisper Service (оффлайн распознавание)

**Движок:** whisper.cpp (Android port)

**Поток:**
```
Запись (MP3) → Валидация → Очередь (Channel) → Whisper → Словарь → БД → Уведомление
```

**Асинхронность:**
- Распознавание в фоне (не блокирует замеры)
- Очередь задач (FIFO)
- Уведомление по завершении

**Запасной вариант:**
- Если whisper.cpp слишком медленный → Vosk (легче, менее точно)
- Если телефон слабый → Сервер (нужен интернет, опционально)

### 3.3. Словарь сокращений

**Где:** В папке проекта (`/Project_XYZ/abbreviations.json`)

**Статус:** ✅ У пользователя уже есть готовый файл

**Формат:**
```json
{
  "parts": {
    "левая петля": "ЛП",
    "правая петля": "ПП",
    "ручка": "Р",
    "замок": "З"
  },
  "defects": {
    "сломана": "слом",
    "царапина": "царап",
    "вмятина": "вмят"
  },
  "rooms": {
    "кухня": "кух",
    "ванная": "ван",
    "спальня": "сп"
  },
  "objects": {
    "дверь": "Д",
    "окно": "ОК"
  }
}
```

**Алгоритм:**
1. Загрузить словарь из `/Project_XYZ/abbreviations.json`
2. Получить raw текст: "левая петля сломана"
3. Применить словарь по категориям (parts → defects → rooms → objects)
4. Получить processed: "ЛП слом"
5. Сохранить в measurement.json

**Автозаполнение:**
- Приложение может предлагать сокращения на основе частоты использования
- Пользователь может добавлять новые сокращения в файл

### 3.4. PDF интеграция (Xodo Intent)

**Как работает:**
1. Приложение открывает PDF в Xodo через Intent
2. Пользователь размечает прямоугольниками (Square annotations)
3. Возвращается в MeasuringAssistant
4. Приложение читает аннотации из PDF (парсинг)
5. Сохраняет слой поверх (JSON с координатами)

**Координаты:** Нормализованные (0-1) — не ломаются при масштабировании

**Библиотеки для парсинга:**
- AndroidPdfViewer
- MuPDF (для чтения аннотаций)

### 3.5. Экспорт

**Форматы:**
- XLSX (основной, через Apache POI или Kotlin библиотеку)
- CSV (простой, для импорта в другие системы)
- JSON (для себя/резерв)

**Важно:**
- Время = timestamp замера, **не экспорта**
- Порядок = хронологический
- Фильтр = только отмеченные на экспорт (`export: true`)

**Пример XLSX:**
```
| ID замера | Объект    | Точка          | Значение | Ед.изм | Время замера       | Заметки (raw)        | Заметки (обр.) |
|-----------|-----------|----------------|----------|--------|--------------------|----------------------|----------------|
| m_001     | D_кух_01  | Ширина (верх)  | 900      | мм     | 2026-04-02 14:30:00| левая петля сломана  | ЛП слом        |
| m_001     | D_кух_01  | Ширина (низ)   | 905      | мм     | 2026-04-02 14:30:00|                      |                |
```

## ЧАСТЬ 4: ПОТОК ДАННЫХ

### 4.1. Поток замера

```
1. Рулетка отправляет измерение по BLE
2. Приложение получает: 900 мм + timestamp
3. Автоматически создаётся measurement_id
4. Сохранение в measurement.json
5. Пользователь видит: "Новый замер: 900 мм"
6. (Опционально) Голосовая заметка → запись → распознавание → словарь → БД
7. (Опционально) Фото → камера → сохранение → БД
8. Экспорт → выбор замеров → XLSX/CSV
```

### 4.2. Поток распознавания аудио

```
1. Пользователь нажимает "🎤 Записать заметку"
2. Запись аудио (MP3, ~5-30 сек)
3. Сохранение: /storage/emulated/0/Android/data/com.measuringassistant/files/media/measurement_001/audio_001.mp3
4. Добавление в очередь (Channel)
5. Whisper распознаёт (фон, ~10-30 сек)
6. Применение словаря сокращений
7. Сохранение transcript в measurement.json
8. Уведомление: "✅ Распознано: ЛП слом"
```

### 4.3. Поток синхронизации

```
1. Android сохраняет в /appdata/media/
2. Syncthing обнаруживает изменения
3. Синхронизация на Windows PC
4. Windows приложение (если будет) читает те же файлы
5. Пост-обработка / экспорт / печать
```
